# Claude Code 프롬프트 작성 지침

> 이 파일은 Claude(claude.ai)가 Claude Code용 프롬프트를 만들 때 매번 참고하는 지침입니다.
> 사용자가 수정 요청을 하면, 이 지침에 맞춰 상세한 프롬프트를 생성합니다.

---

## 1. 프롬프트 필수 구성요소

모든 프롬프트에 아래 섹션을 빠짐없이 포함할 것:

### ■ 현재 상태
- 지금 어떤 상태인지 구체적으로 설명
- 문제가 있다면 증상을 정확히 기술
- 스크린샷에서 확인된 내용 반영

### ■ 변경 요청
- 원하는 결과를 명확하게 기술
- 애매한 표현 금지 (예: "적절히" "비슷하게" → 구체적 수치/참조 대상 명시)

### ■ 상세 요구사항
- 항목별로 번호를 매겨 상세하게 작성
- 디자인 변경 시: 참조 대상(어떤 탭의 어떤 요소)을 명시
- 기능 변경 시: 입력→처리→출력 흐름을 명시
- 데이터 변경 시: 어떤 필드가 어떤 값으로 바뀌는지 명시

### ■ 수정 대상 파일
- 수정할 파일 경로를 명시적으로 나열
- 신규 파일이면 "신규" 표시

### ■ 절대 건드리지 않을 것
- 수정 범위 밖의 파일/함수/로직을 구체적으로 나열
- "다른 코드"처럼 모호하게 쓰지 말고 구체적으로

### ■ 작업 순서
- 1단계부터 순서대로 기술
- "먼저 기존 코드를 읽어서 확인" 단계를 항상 포함
- 마지막에 "빌드 테스트" + "git push" 필수 포함

### ■ 검증 체크리스트
- 수정 완료 후 확인할 항목을 체크리스트로 작성
- 기존 기능이 깨지지 않았는지 확인 항목 포함

## 2. 프로젝트 핵심 규칙 (매 프롬프트에 암묵적으로 적용)

### ⚠️ 최우선 규칙 (절대 위반 금지)
1. **이해 안 되면 무조건 질문**: 만들기 전에 이해가 안 되거나 확실하게 하고 싶은 건 무조건 사용자에게 다시 질문할 것. 추측해서 진행 금지.
2. **수정 요청한 것만 수정**: 다른 파일/로직은 절대 건드리거나 변경 금지. 1줄도 안 됨.
3. **하나씩 확인하며 진행**: 작업이 여러 개면 하나 완료될 때마다 사용자가 확인할 수 있게 체크하며 진행. 누락 방지.
4. **기존 데이터 보존**: localStorage에 저장된 기존 데이터(제품, 재고, 프로모션, 설정, 발주이력 등)는 절대 삭제/초기화/변경 금지.
5. **UI 변경 시 이미지 먼저**: UI 변경 요청 시 항상 이미지(목업)를 먼저 만들어서 보여준 후 진행.
6. **연동 작업은 알아서 완료**: Vercel, Supabase, 경영박사, 네이버 등 연동 가능한 건 전부 알아서 작업하고 결과를 보고할 것.

### 일반 규칙
7. public/manager/ 파일 수정 시 기존 UI/기능 깨뜨리지 않을 것.
8. 코드는 JavaScript로 작성.
9. API 키는 코드에 직접 입력 금지 → 환경변수 사용.
10. 에러 처리(try/catch + 로그) 반드시 포함.
11. 경영박사 API는 반드시 SOAP POST 방식.
12. 응답은 한국어로.
13. 디자인 참조 시 임의로 색상/스타일 변경 금지. 참조 대상과 100% 동일하게.
14. 완료 후 반드시 git push.

## 3. 프로젝트 구조

```
daehan-platform/
├── public/manager/          ← 기존 HTML (원본 보존, 수정은 여기서)
│   ├── index.html           ← UI 구조
│   ├── style.css            ← 스타일 (거의 수정 안 함)
│   └── app.js               ← 전체 로직 (5,400줄)
├── src/app/                 ← Next.js 페이지/API
│   ├── api/erp/             ← 경영박사 API Routes (서버사이드)
│   ├── api/naver/           ← 네이버 API Routes (서버사이드)
│   └── 각 페이지/           ← React 버전 (나중에 사용)
├── src/lib/
│   ├── erp.ts               ← 경영박사 SOAP 공통 함수
│   └── supabase.ts          ← Supabase 클라이언트
├── docs/                    ← API 문서
│   ├── 경영박사_API.md
│   └── 네이버_커머스_API.md
├── CLAUDE.md                ← Claude Code 자동 참조 지침
└── .env.local               ← 환경변수 (git 제외)
```

## 7. 프롬프트 품질 체크리스트

프롬프트를 사용자에게 전달하기 전, 아래 항목을 자체 검증:

### 최우선 체크
- [ ] 요청 내용이 애매한 부분이 있으면 프롬프트 만들기 전에 사용자에게 질문했는가?
- [ ] UI 변경이 포함되면 이미지(목업)를 먼저 보여줬는가?
- [ ] 여러 작업이면 하나씩 확인하며 진행하도록 단계가 나뉘어 있는가?

### 프롬프트 구성 체크
- [ ] 현재 상태가 구체적으로 기술되었는가?
- [ ] 변경 요청이 명확한가? (애매한 표현 없는가?)
- [ ] 상세 요구사항이 항목별로 번호 매겨져 있는가?
- [ ] 디자인 참조 대상이 있으면 "먼저 읽어서 확인" 단계가 포함되었는가?
- [ ] 수정 대상 파일이 명시되었는가?
- [ ] "절대 건드리지 않을 것" 섹션이 구체적인가?
- [ ] 작업 순서가 단계별로 기술되었는가?
- [ ] "기존 코드 먼저 읽기" 단계가 포함되었는가?
- [ ] "빌드 테스트" + "git push" 가 마지막에 있는가?
- [ ] 프롬프트가 너무 간략하지 않은가? (최소 20줄 이상)
- [ ] 임의 변경을 방지하는 제약 조건이 충분한가?
- [ ] 검증 체크리스트가 포함되었는가?
- [ ] Vercel/Supabase 연동이 필요한 부분은 포함되었는가?

## 8. 자주 하는 실수 방지

### Claude Code가 자주 하는 실수:
1. ❌ 요청 안 한 파일까지 "개선"한다며 수정
2. ❌ 디자인 참조 시 색상/스타일을 임의로 변경
3. ❌ 기존 함수 로직을 "리팩토링"한다며 변경
4. ❌ 데이터 구조를 "최적화"한다며 변경
5. ❌ CSS 클래스를 Tailwind로 임의 변환
6. ❌ 한 번에 너무 많은 파일을 수정
7. ❌ 빌드 테스트 없이 push

### 방지 문구 (프롬프트에 포함):
- "다른 로직은 1줄도 수정하지 마"
- "기존 코드를 먼저 읽어서 확인하고 시작해"
- "버튼/색상/스타일을 임의로 바꾸지 마. [참조대상]과 100% 동일하게"
- "수정 전후 diff 확인 — 요청한 부분 외에 변경된 줄이 있으면 되돌리기"
